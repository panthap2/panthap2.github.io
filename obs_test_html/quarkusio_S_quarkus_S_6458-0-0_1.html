I'm trying to implement second-level caching with Panache but @ManyToOne relations seem not to be cached.

E.g. when having Continent / Country entities likes:

@entity
@Cacheable
public class Continent extends BasePanacheEntity {

@Id
public String id;
public String code;
public String name;

}

@entity
@Cacheable
public class Country extends BasePanacheEntity {

@Id
public String id;
public String alpha2Code;
public String alpha3Code;
public String name;

@ManyToOne(fetch = FetchType.LAZY)
// Note, also tried adding this: @Cache(usage = CacheConcurrencyStrategy.READ_ONLY)
@JoinColumn(name = "continent_id")
public Continent continent;

}

When I the first time query a Country and access the Continent the following queries are executed (as expected):

Hibernate:

continent0_.id=?

The next time it only executes:

Hibernate:
select
continent0_.id as id1_3_0_,

I also tried adding @Cache(usage = CacheConcurrencyStrategy.READ_ONLY) on the relation but that didn't make any difference.

My cache properties look like:

Maybe I'm doing something wrong or can these relations not be cached this way?
When I query the Continent directly as root object, then it is cached when I again query the same Continent as root object btw. But via the Country it still does the query.

I basically want to cache a lot of these kind of entities as they simple static reference data.
And many other entities refer to these entities, I don't want to either re-fetch them every time, nor use eager fetching.

So the problem is with "Enhanced Proxies": it's a new feature in Hibernate ORM, which is enabled in Quarkus.

It seems to consistently bypasss the 2LC in org.hibernate.persister.entity.AbstractEntityPersisterinitializeEnhancedEntityUsedAsProxy. Since the specific trigger is a field read operation triggering the lazy load I wasn't seeing this in a vanilla ORM test - as the field read fashion is Panache specific.

I'll see to get a fix in ORM, as Enhanced Proxy is otherwise super useful.

Sanne commented May 13, 2020

@Sanne This was introduced in Hibernate 5.4.14? That could explain why it was working correctly in Spring project. Probably that version of Spring Boot was using another Hibernate version.

sorry, I missed the question. No the feature was introduced approximatley a year ago, but it needs to be enabled explicitly.

Let me share a bit of context.

While tuning is interesting, over the years Hibernate ORM has grown a massive amount of configuration properties; this is becoming not only a usability pain but also a maintenance problem as we need to test various features in combination with each others permutations. I'm therefore figthing hard to try reducing configuration knobs :)

Most often we introduce new features together with a kill switch for backwards compatibility reasons; sometimes to keep a novel experimental new idea from causing too many problems it will be turned off by default, unless you know the magic properties to set..

In Quarkus we can do higher automation of configuration because of the application analysis phase, and because of it being in the lucky position currently of focusing on new applications, we have the luxury to enable really cool stuff by default - and remove/hide the configuration knobs.

Enhanced Proxies is one such thing; it's now enabled by default in Quarkus as it has no drawbacks - other than being relatively less tested code, like you noticed.

In such cases though I prefer digging and fixing the problem rather than giving people the configuration property to turn it off; it's a battle to make sure the whole platform will work fine with least trouble, all pieces combined properly.

I hope that makes sense!

HHH-14004 was fixed; I'll leave this issue open as we need to release it and upgrade Quarkus.
