I do not have much experience in ByteBuddy, just started this week. Please bear with me. I searched the code base (especially tests) and issue list (also closed ones) here as well as StackOverflow. My situation is this:

I am instrumenting JRE class String using an Advice and a transformator attached during runtime via ByteBuddyAgent. The advice as such works fine, so there are no problems with redefining the bootstrap class.

Now I want to generalise my advice, using it as a wrapper and dispatching to helper methods in other classes and it works 99% fine. There is just one edge case which I cannot get working:

  • My @OnMethodEnter advice has one parameter @AllArguments(readOnly = false, typing = DYNAMIC) Object[] args which I forward to the helper method in the other class.
  • If I change any value in the arguments array e.g. via args[0] = newValue, the change is not reflected in the advice after the helper method returns to it, i.e. the helper method cannot change the parameter list before the wrapped method is called after exiting the advice.
  • Besides, even if I change the value in the advice method itself, this does not work. I really have to create new array via args = new Object[] { newValue, args[1] }; if I want to manipulate the method parameters before proceeding to the target method.
  • But even that approach does not work in a helper class. I could not find a way to reflect the arguments changes back into the actual static advice method and from there on to the target method.

One notable problem I had was making the helper class known to the bootstrap classloader. I solved that via something like this:

I also had to put both the byte-buddy and byte-buddy-agent dependencies on the boot classpath when running this from a JUnit test. Strangely, when running from an application class and not from a test class, I do not need to change the boot classpath.

But as I said, the program is running, the instrumentation happening and the advice does what it should and can even communicate data on to the corresponding @OnMethodExit advice. The only thing not working is the @AllArguments part.

Normally when I manipulate entries of an Object[] (or whatever type of array) in a called method, the calling method can see the changes. Is this some kind of problem with the helper class being resolved again in the boot classloader? Why can I communicate the array into the helper method but not back out?

raphw commented Apr 29, 2020

So, advice is a nice trick but you do indeed need to assign the entire array to make this work. Here is why:

Byte Buddy finds byte code locations where you access an annotated variable. It then replaces the location with an instruction that represents what the annotation indicates. So if you do Object o = args, then Byte Buddy reads all parameters into an array and replaces args by it. If you write args[0], in byte code this is two instructions. Again, Byte Buddy replaces the args bit but after being read, the array is a value without attachment to the original arguments.

The only way for setting an argument is by assigning args a value where Byte Buddy can replace this byte code instruction. Again, the array object is not tagged in any particular way.

I came up with two approaches:

  1. Copy args into a SortedMap<Integer, Object> with the keys being the original array indexes. Hand the map over to the helper method which can easily use argsMap.put(index, newValue) in order to update the map. After returning, copy the map contents into a newly created Object[] returnedArgs and finally issuing args = returnedArgs. This works, even if it is a bit ugly with loops and all.
  2. Just use Object[] argsCopy = Arrays.copyOf(args, args.length), hand the argsCopy over to the helper method (which can keep the Object[] args signature and does not need to use a map), let the helper manipulate the copied array and finally assign args = argsCopy in order to make ByteBuddy notice that the argument list was changed and it should do its magic.

Both approaches work in my real code manipulating JRE method String.replaceAll. But I was very surprised to find the array copying approach failing in my stripped down minimal example from above. If I change DemoAspect to:

Update: Another workaround which does not yield a VerifyError during transformation is this:

package de.scrum_master.agent.bytebuddy;

import net.bytebuddy.asm.Advice;

import java.util.Arrays;

import static net.bytebuddy.implementation.bytecode.assign.Assigner.Typing.DYNAMIC;

public class DemoAspect {
  @Advice.OnMethodEnter
  public static String before(
    @Advice.AllArguments(readOnly = false, typing = DYNAMIC) Object[] args
  ) {
    System.out.println("args = " + args + " -> " + Arrays.deepToString(args));
    Object[] argsCopy = new Object[args.length];
    System.arraycopy(args, 0, argsCopy, 0, args.length);
    // This leads to a VerifyError during transformation
//    Object[] argsCopy = Arrays.copyOf(args, args.length);
    System.out.println("argsCopy = " + argsCopy + " -> " + Arrays.deepToString(argsCopy));
    new AdviceHelper().before(argsCopy);
    System.out.println("argsCopy = " + argsCopy + " -> " + Arrays.deepToString(argsCopy));
    args = argsCopy;
    System.out.println("args = " + args + " -> " + Arrays.deepToString(args));
    return "proceed";
  }
}
