Summary of problem

During Annotation handling at a high parallelism getting different unexpected exceptions.
Worst case scenario is actually when we don't get an exceptions and the annotations are not being processed correctly which can cause data loss!

One of the exceptions is ClassCastException on the first return statement which is easy to see it is probably not a coding issue:

public <A extends Annotation> A getAnnotation(Class<A> annotationType) {
    checkNotNull(annotationType);
    for (Annotation annotation : annotations) {
      if (annotationType.isInstance(annotation)) {
        return annotationType.cast(annotation);
      }
    }
    return null;
}

Reducing the parallelism of the code reduces the chances of failures.
When removing the parallelism of the code we can't reproduce.
There are many other exceptions that happens on the original code but I believe they are all the same root cause.

Diagnostic files

No crashes - just exceptions.

r30shah commented May 6, 2020

Found the problem. Summarizing the issue here.

An instanceOf node in OpenJ9 has generally two children, first one is the object we are checking (Here it is annotation) and second one is the Class (CastClass) with which we are checking relation of this object (Here it is Class of annotationType).
In this case, the CastClass is runtime variable, so on runtime we load the class of the annotationType and depending on type of it (Regular Class/Interface, etc) we execute runtime tests. If we find that annotationType is Interface at runtime, we do not have much options in terms of inlined fast tests to avoid a slow JIT helper call to get the results. For this reason on IBM Z we have added a onsite (at that particular instanceOf site) cache to store the result of previous helper call, which we would check next time, if it fails all the inline tests and is about to make a slow helper call.
Problem occurred in this highly paralleled work-load where I am assuming, it was executing the same instanceOf site in more than 500 threads, which would have caused a race condition in caching the results (We store J9Class of both object and CastClass and next time check both slot). This race condition would have caused unexpected data in the cache looking at which it would have taken wrong decision (Returning true when annotation is not instanceOf annotationType).

So problem was that we were loading/storing associated castClass and Object Class separately from the memory, which for 64-Bit JVM should be done so that it is quadword block concurrent. I have tested a fix that uses LPQ and STPQ which loads and stores from the memory in quadword block boundary and do not see the failure.
Working on polishing this fix, creating a PR soon
