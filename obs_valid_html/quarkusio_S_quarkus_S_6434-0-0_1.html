loicmathieu commented Jan 7, 2020

OK, it's a tricky one! The thread ID is used instead of the ThreadName because the thread that generates the event is no longuer available. It's a build thread.
So packaging the application and initializing the index from it will make the trick (native will also works as we have a substitution).
Or initializing the index from an application that didn't include Flyway as it's the flyway extension that creates log messages during the build, and those log messages generates the wrong index.

@dmlloyd long story short: there is a tricky issue where a log event created by Flyway during the build process in dev mode ends up having the TreadName the same as the ThreadID (in dev mode the substitution is not used so it uses JMX to find the ThreadName from the ID). If I recall correctly, build log events are replayed at runtime in dev mode, at this time the Thread that generates the event is no longuer available. I don't think there is an easy way to workaround the issue other than waiting for mp911de/logstash-gelf217 ?
